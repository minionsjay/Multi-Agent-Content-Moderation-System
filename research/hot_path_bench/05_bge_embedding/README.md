# 05 BGE Embedding (BAAI/bge-small-zh-v1.5)

## 测试什么

BGE-small-zh-v1.5 是热路径中**唯一调用神经网络模型**的环节，也是**唯一的瓶颈**。它负责将文本转为 512 维向量，供 ChromaDB 做语义相似度检索。整个热路径约 80% 的延迟和 100% 的计算密集度都集中在这里。

## 测试项

### Test 1: 单文本延迟（模型预热后）

| 文本长度 | P50 | P99 | Avg |
|------|------|------|------|
| 短 (7 chars) | 0.00ms | 9.47ms | 0.19ms |
| 中 (20 chars) | 0.00ms | 11.47ms | 0.23ms |
| 长 (500 chars) | 0.00ms | 39.89ms | 0.80ms |

**关键发现**：P50 为 0.00ms 是因为 Embedding 缓存命中后直接返回（内存操作），不走 BGE 推理。P99 的尖峰（9-40ms）是缓存 miss 时的真实 BGE 推理时间。**平均延迟很低是因为缓存拦截了大量重复文本**。

### Test 2: 批量推理 vs Batch Size

同一批文本，不同 batch size 下的吞吐。

| Batch Size | 总延迟 | 单文本延迟 | 等效 QPS |
|------|------|------|------|
| 1 | 0.6ms | 0.61ms | **1,637** |
| 4 | 0.7ms | 0.18ms | **5,566** |
| 8 | 0.7ms | 0.09ms | **10,937** |
| 16 | 1.0ms | 0.07ms | **15,338** |
| 32 | 1.5ms | 0.05ms | **21,167** |
| 64 | 2.6ms | 0.04ms | **24,901** |
| 128 | 5.1ms | 0.04ms | **25,290** |

**关键发现**：

- batch=1 → batch=32：**13 倍吞吐提升**（1,637 → 21,167 QPS）
- batch=32 → batch=64：边际收益下降（21K → 25K，只有 1.2x）
- **batch=32 是最佳甜点**：单文本 0.05ms，总延迟 1.5ms，吞吐 21K QPS

为什么会这样？Transformer 的前向传播中，矩阵乘法占据了 70%+ 的计算时间。批量输入让矩阵维度变成 [batch_size, seq_len, hidden_dim]，可以利用 CPU 的 SIMD 向量化指令（AVX/SSE）并行计算。

### Test 3: Embedding 缓存效果

模拟真实流量（40% 独特 + 60% 重复），1000 条文本。

| 指标 | 值 |
|------|------|
| 总文本 | 1,000 |
| 总耗时 | 4.5s |
| Embedding 缓存命中率 | **91.4%** |
| 有效吞吐 | 225 texts/sec |

**结论**：有了 Embedding 缓存，91% 的重复文本不需要重新推理。对于日均 5000 万条的流量，假设 30% 是重复文本，缓存可以节省约 1500 万次 BGE 推理调用。

### Test 4: 向量属性

| 属性 | 值 |
|------|------|
| 维度 | 512 |
| 每向量内存 | 2,048 bytes (2 KB) |
| 100 万向量内存 | ~1,953 MB (2 GB) |
| L2 范数 | 1.0000 (已归一化) |
| 值范围 | [-0.31, +0.24] |
| 非零元素 | 512/512 (密集向量) |

**结论**：512 维 × float32 = 2KB/向量。对于 ChromaDB 缓存，100 万条记录约占用 2GB 内存（仅向量部分）。

## 实际意义

BGE Embedding 是热路径中唯一需要扩容的环节。好消息是：
1. **Embedding 缓存**（已实装）消除 60-90% 的重复推理
2. **批量推理**可将单核吞吐从 1.6K 提升到 21K QPS
3. **ONNX 导出**可进一步将单次推理从 0.6ms 降到 0.2ms

三者叠加，单核 BGE 吞吐可以从 200 QPS 提升到 **~5,000 QPS**，足够覆盖日均 5 亿条的流量。
