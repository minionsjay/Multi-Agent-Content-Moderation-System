# 02 AC Automaton Keyword Matching (pyahocorasick)

## 测试什么

Gateway 热路径的第二层：用 Aho-Corasick 自动机在线性时间内扫描文本中的所有关键词。AC 自动机预先将所有关键词构建成一个 Trie + 失败转移边，扫描时每个字符只走一次状态转移。

## 为什么选 AC 自动机而不是正则

正则的 `word1|word2|word3|...` 在每个字符位置都要回溯尝试所有 alternation，复杂度 O(n × k)。AC 自动机的复杂度是 O(n)，与词库大小无关。

## 测试项

### Test 1: 延迟 vs 文本长度

用当前默认词库（约 50 个关键词），扫描不同长度的随机文本。首次调用包含 jieba 词典加载（~500ms）。

**结果**（每 1000 次调用的总耗时）：

| 文本长度 | 1000 次耗时 | 单次延迟 |
|------|------|------|
| 20 chars | 756ms | 756μs |
| 50 chars | 212ms | 212μs |
| 200 chars | 881ms | 881μs |
| 500 chars | 2076ms | 2076μs |
| 1000 chars | 4394ms | 4394μs |
| 5000 chars | 23286ms | 23286μs |

**结论**：延迟与文本长度成线性关系（O(n)）。200 字的中文评论约 0.9ms/次，1000 QPS 场景下 AC 自动机不是瓶颈。注意：这里的主要开销不在 AC 自动机（C 扩展，释放 GIL），而在 jieba 分词（Python 层）。

### Test 2: 延迟 vs 词库大小

构建不同大小的自动机（10 → 10,000 词），在同一个文本上运行 5000 次。

**结果**：

| 词库大小 | 5000 次耗时 | 单次延迟 |
|------|------|------|
| 10 | 1.15ms | 0.23μs |
| 50 | 1.14ms | 0.23μs |
| 100 | 1.18ms | 0.24μs |
| 500 | 1.19ms | 0.24μs |
| 1,000 | 1.20ms | 0.24μs |
| 5,000 | 1.20ms | 0.24μs |
| 10,000 | 1.20ms | 0.24μs |

**结论**：O(1) 与词库大小无关！即使词库扩展到 10,000 个词，扫描延迟完全不变。这是 AC 自动机的核心优势 —— 再多关键词也不影响速度。

### Test 3: AC 自动机 vs 暴力匹配

同一个文本（~1000 字），不同词库大小，对比 AC 自动机和 `if word in text` 循环。

| 词数 | AC 耗时 | 暴力耗时 | AC 加速比 |
|------|------|------|------|
| 10 | 3.14ms | 1.61ms | 0.5x（小词库暴力更快） |
| 50 | 3.10ms | 7.79ms | **2.5x** |
| 100 | 3.14ms | 15.34ms | **4.9x** |
| 500 | 3.11ms | 77.85ms | **25.0x** |

**关键拐点**：当词库小于 10 个时，暴力匹配反而更快（没有构建自动机的开销）。词库超过 20 个后，AC 自动机开始领先。词库越大，优势越大。

### Test 4: 真实关键词匹配

用当前系统内置的默认词库测试 10 种典型场景：

| 测试文本 | 检测标签 | 置信度 | 延迟 |
|------|------|------|------|
| "今天天气真好适合出去玩" | none | 0.0 | ~1μs |
| "你真是个傻逼什么都不懂" | toxic | 1.0 | ~200μs |
| "支持台独台湾是一个独立国家" | politics | 1.0 | ~200μs |
| "我要杀了你这个混蛋" | violence | 1.0 | ~200μs |
| "you are a fucking idiot and stupid" | toxic | 1.0 | ~200μs |
| "操场上同学们在操练队列" | none | 0.0 | ~700μs |
| "加我微信看裸照约炮一夜情" | porn | 1.0 | ~200μs |
| "真人百家乐在线赌博日赚千元" | gambling | 1.0 | ~200μs |

**结论**：正常文本 < 1μs 返回，违规文本 ~200μs 检测。jieba 分词是主要开销（"操场上"的 whitelist 匹配需要 jieba 分词 + 正则白名单 + 位置验证，所以更慢）。

## 实际意义

关键词匹配是热路径中拦截率最高的层（预期 20-25%）。在日均 5000 万条的流量中，约 1000-1250 万条在 < 1ms 内被拦截，**成本为零，不调用任何模型**。
