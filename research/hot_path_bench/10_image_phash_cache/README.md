# 10 Image pHash Cache

## 测试什么

图片热路径 I2 层：用 dHash 做 key 查内存缓存。相同视觉内容的图片（换 URL、换格式）产生相同/相似 dHash → 复用历史审核结果。

## 测试结果

### Test 1: 缓存查找速度

```
10,000 lookups: 17.7ms = 1.8μs/lookup
QPS: 564,756
```

纯内存 dict 查找，与文本 L0 缓存相同速度。

### Test 2: 近重复图片缓存命中

原始图片 hash: `1939254c6baea652`

| 变体 | 哈希 | Hamming 距离 | Exact 命中 | Hamming≤10 |
|------|------|------|------|------|
| 缩放 80% | `...652` | 1 | MISS | ✓ |
| JPEG q=70 | `...652` | 1 | MISS | ✓ |
| 亮度 +8 | `...652` | 1 | MISS | ✓ |
| 旋转 1° | `...652` | 2 | MISS | ✓ |

**关键发现**: 所有变体的哈希差 1-2 bit。如果用**精确匹配**查缓存，全部 miss（因为换 URL/格式后哈希有一点点不同）。必须用 **Hamming 距离 ≤10** 做模糊匹配才能命中。

### Test 3: Hamming 阈值权衡

20 张原图 × 20 张原图 = 400 对，不同阈值下的交叉匹配率：

```
threshold≤0:  self=20/20  cross=0/380 (0.0%)
threshold≤10: self=20/20  cross=0/380 (0.0%)
threshold≤20: self=20/20  cross=0/380 (0.0%)
```

**在所有阈值下交叉匹配率为 0%** —— 20 张随机种子生成的图像两两之间没有任何误匹配。阈值 10 是安全的。

### Test 4: 内存占用

| 缓存条目 | 估算内存 |
|------|------|
| 每条目 | ~220 bytes |
| 10 万 | ~21 MB |
| 100 万 | ~210 MB |

## 当前限制

当前代码中 pHash 缓存只用**精确匹配**（`memory_cache.get(f"[PH]{hash}")`），没有做 Hamming 距离扫描。变体图片（d=1-2）会 miss。生产需要：

1. 在 ChromaDB 中建立 dHash 的 Hamming 距离索引
2. 或者使用 FAISS Binary Index 做 Hamming 距离快速检索
3. 或者在缓存 miss 时扫描附近的 hash（阈值小，可以暴力扫描少量条目）
