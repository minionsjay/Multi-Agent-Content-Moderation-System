# 01 L0 Memory Cache (SHA256 + TTLCache)

## 测试什么

Gateway 热路径的第一层：内存里的精确匹配缓存。文本先做 SHA256 哈希，再查 Python dict (TTLCache)。完全相同的文本之前审过的话，直接返回缓存结果，跳过后面所有步骤。

## 为什么是热路径第一层

因为最快 —— 纯内存操作，没有 I/O，没有模型推理。而且很多内容审核场景中，完全重复的文本非常常见（垃圾广告批量刷、同一用户多次提交相同内容）。

## 测试项

### Test 1: 延迟 vs 缓存大小

缓存从 1K 到 500K 条，每个做 10 万次查询。

**结果**：

| 缓存大小 | 10 万次查询耗时 | 单次延迟 | 理论 QPS |
|----------|------|------|------|
| 1,000 | 104.4ms | 1.04μs | 957,454 |
| 10,000 | 109.4ms | 1.09μs | 914,238 |
| 50,000 | 125.6ms | 1.26μs | 796,375 |
| 100,000 | 132.6ms | 1.33μs | 754,221 |
| 500,000 | 135.3ms | 1.35μs | 739,018 |

**结论**：TTLCache 的 dict 查找是 O(1)，规模从 1K 到 500K 只增加 30% 延迟。即使是最坏情况（1.35μs），也能跑 74 万 QPS。**不是瓶颈，永远不是。**

### Test 2: 哈希算法开销

比较 SHA256、MD5、Python hash() 三种哈希，10 万次哈希计算。

**结果**：

| 算法 | 10 万次耗时 | 单次延迟 |
|------|------|------|
| Python hash | 10.6ms | 0.11μs |
| SHA256 | 124.4ms | 1.24μs |
| MD5 | 186.5ms | 1.87μs |

**结论**：SHA256 每次 1.24μs，比 dict 查找本身（1.35μs）还快一点。哈希 + 查找合计约 2.6μs。MD5 反而更慢（因为 Python 的 hashlib 对 SHA256 有硬件加速）。选择 SHA256 是因为碰撞概率 < 10^-60，对于内容审核系统来说，哈希碰撞意味着一个违规文本可能被错误匹配为安全文本，不可接受。

### Test 3: 内存占用估算

| 缓存条目 | 估算内存 |
|------|------|
| 10,000 | ~4 MB |
| 50,000 | ~21 MB |
| 100,000 | ~42 MB |
| 500,000 | ~208 MB |

### Test 4: 原始吞吐量

50 万次混合读写操作（67% 读 + 33% 写），总耗时 1340ms = **373K ops/sec**。

### Test 5: SHA256 碰撞检测

10 万条相似文本（每个只差 1 个字符）→ SHA256 碰撞数为 0（测试中的"碰撞"是文本本身相同导致的，不是 SHA256 哈希碰撞）。

## 实际意义

在日均 5000 万条的场景下，其中 5-15% 是完全重复的（同一广告反复发）。L0 缓存可以在 < 3μs 内拦截这些请求，**等效免费获得了 5-15% 的吞吐提升**。
