# 04 ChromaDB Semantic Cache

## 测试什么

Gateway 热路径的第三层：语义缓存。当前两层（内存缓存 + 关键词）都解决不了的情况 —— 文本不重复、不含关键词，但**意思跟以前审过的某条违规文本很像**。ChromaDB 存储所有历史审核结果的 embedding，用 cosine 相似度检索。

## 测试项

### Test 1: 查找延迟

对已有 18 条记录的 collection 做 100 次查找（不包括 embedding 时间）。

**结果**：

| 指标 | 延迟 |
|------|------|
| P50 | 1.10ms |
| P99 | 2.73ms |
| Avg | 1.13ms |

**结论**：ChromaDB 查询（不包括 embedding）约 1ms，对于缓存层可接受。加上 BGE embedding 的 ~0.2ms（已预热），总延迟约 1.3ms。

### Test 2: 写入延迟

20 次写入（存储 embedding + metadata）。

**结果**：

| 指标 | 延迟 |
|------|------|
| P50 | 14.82ms |
| P99 | 19.79ms |
| Avg | 15.35ms |

**结论**：写入比读取慢 13 倍。HNSW 图插入需要更新多层邻居表，比查询昂贵很多。在高并发场景下，**写入可能成为瓶颈**（尤其是冷路径判决后的写回操作）。

### Test 3: 相似度阈值边界

测试不同程度相似的文本是否能命中缓存（阈值 0.95）。

| 文本 | 相似度 | 结果 |
|------|------|------|
| 完全相同的文本 | 1.0000 | HIT |
| 只改一个字（"需要"→"必须"） | 0.9950 | HIT |
| 意思相近（更短，用词不同） | 0.0000 | MISS |
| 完全不同-安全内容 | 0.0000 | MISS |
| 完全不同-违规内容 | 0.0000 | MISS |
| 短文本 "违规" | 0.0000 | MISS |

**发现**：BGE-small-zh 的 cosine 相似度呈**两极分化** —— 完全一致或微调个别字 → 0.99+，语义不同 → 0.00。这是因为归一化后的高维向量（512 维）在高维空间中，大多数随机向量都近似正交（cos ≈ 0）。只有真正语义接近的文本才能得到高相似度。

这也意味着：**当前阈值 0.95 是合理的**，不太可能误匹配。

### Test 4: 读写吞吐对比

| 操作 | QPS | 延迟 |
|------|------|------|
| Read | **956** | 1.05ms avg |
| Write | **36** | 27.64ms avg |
| Read/Write 比 | 26.4x | - |

**结论**：读远快于写。在日均 5000 万条的场景下，假设 20% 走冷路径且结果需要写回 ChromaDB，那就是每天 1000 万次写入 = 115 QPS。**单实例写不入**。解决方案：
1. 异步攒批写入（32 条一批，batch insert）
2. ChromaDB server 模式（多线程写入）
3. 不需要实时的写入，可以在 Action agent 中异步完成

## 实际意义

ChromaDB 缓存是 Gateway 三层中唯一需要"学习"的层 —— 用得越多，缓存越丰富，命中率越高。冷启动期间命中率 < 5%，运行几个月后预期达到 30-40%。
